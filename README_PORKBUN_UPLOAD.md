# Nik Murano Professional Showcase - Project-page upload

Upload the contents of this folder to the web root for nikmurano.com.

This package is intentionally in-page project-surface first:
- `index.html` is the standalone professional showcase page.
- Screenshot/link-out evidence modules have been replaced with in-page HTML/CSS work-surface modules.
- There is no `live-work/` subsite and no old portfolio interface bundled here.
- Top navigation opens internal project pages by hash.
- Current generated coverage is 241 internal sections: project-surface sections, Master Bundle briefing pages, inherited showcase pages, plus the full project index and source atlas.
- Every project section carries a reviewer layer with built-state discipline, source-trace claim boundary, load-bearing source-utilization plan, and representative APA 7 lineage.
- `project_blueprints/` is the maintainable source-of-truth layer for project vision, mechanism, built state, source use, plan, status, and remaining gaps.
- Every project blueprint folder includes a `README.md` and `blueprint.json`.
- `proof_builds/`, `proof_demos/`, and `proof_exhibits/` provide the proof-engineering layer: proof packets, demo protocols, executable flagship exhibits, and an all-component proof-protocol runner covering 3,639 component cases.
- `proof_exhibits/source_strengthening.html` adds the local book-source strengthening layer. It routes the indexed Kindle-PDF library into proof roles while explicitly warning that metadata routing is not a substitute for close reading before quotation or citation.
- `proof_receipts/` provides the receipt-level proof layer for all 3,639 component cases. It distinguishes package-verified proof receipts from direct local artifact traces and states the falsification rule, claim ceiling, and next strengthening action for each component.
- `room_engine_operationalization/` adds a runnable package-level engine slice for every project: each project has named instruments, a fixture, acceptance tests, an export schema, and a receipt while preserving the boundary that external full-system completion remains a separate claim.
- `room_runtime_kernel/` adds the shared stateful runtime for those room-engine slices: project selection, instrument execution, local state persistence, acceptance checking, and receipt export.
- `room_project_workbenches/` adds one direct executable workbench page per project, each with project-specific instruments, local state, acceptance checks, and receipt export.
- `full_operational_build_program/` promotes proof builds into full-build contracts. It defines what operational completion means for each project type, what minimum release artifact must be built, which sources still govern the build, and which tests must pass before stronger claims are allowed.
- `full_build_control_plane/` promotes proof builds into operational build governance. It classifies each project by final object type, runtime target, source requirement, test suite, build tranche, and first full-build step so software platforms, research projects, manuscript systems, source archives, and public rooms are not confused.
- `component_proof_builds/` provides the component proof-build layer. It converts all 3,639 component proof rows into artifact specifications, inspection protocols, falsification rules, source roles, and next-build actions, plus one distinctiveness build plan per project.
- `source_close_read_extraction/` expands source utilization from routing into machine-assisted text extraction, source-derived concept cues, and project-specific bolstering duties across the full source-application matrix.
- `source_utilization_depth_audit/` distinguishes broad book-source coverage from true close-read utilization. It classifies source applications from L1 source assignment through L6 human-verified artifact integration, adds short quote candidates as selection cues, and states plainly that the package is not source-maxed until passages are verified and integrated.
- `source_passage_utilization/` creates passage-level close-read contracts that locate candidate source files, bind each source to proof capsules, and state which output, test, and claim boundary must change before stronger source claims are allowed.
- `source_application_control_plane/` converts sources into project machinery: source roles, project-specific applications, outputs changed, tests governed, role gaps, and close-read tasks.
- `source_to_build_bindings/` connects routed source candidates to operational build modules, tests, design decisions, claim ceilings, and bibliographic hygiene tasks so sources strengthen the system rather than remaining decorative bibliography.
- `source_workbench/` provides the broad source-routing layer. It assigns source roles to all 243 projects and 3,639 component proof rows; `source_passage_utilization/` then converts the active close-read subset into passage-level project-change contracts.
- This is a no-image, no-assets release package designed to remain far below the 40 MB upload limit.

Public packaging rule: this upload folder should not expose local workstation paths, private source directories, screenshots, or heavyweight artwork.
