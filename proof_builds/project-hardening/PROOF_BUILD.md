# Proof Build: Quality Review — checking project claims against evidence, scope, and readiness

**Project ID.** `project-hardening`

**Proof tier.** authored-proof-build-needs-demo

**Public claim.** A quality-review and hardening surface for moving projects toward release.

**Distinctive mechanism.** It asks what is built, what claim it supports, what evidence proves it, what can break, and what demonstration would make the claim stronger, moving projects through Stage S0 Prospecting Build, Stage S1 Survey, Stage S2 Lite, Deep Anchored work, Stage S3 SOPs, red-team executable SOPs, Stage S4 Yield, Stage S5 Pilot Pre-register, RETIRE Validation, committee-facing S3 readiness, release freeze, redundancy review, and sign-off.

**Use value.** It is the anti-overclaim room: it turns ambition into staged proof obligations, release criteria, and review gates.

**Minimum public demo.** select one overbroad claim and show the exact reason it is kept, narrowed, deferred, or removed

**Claim ceiling.** The page may claim proof-structured components and visible verification targets. It may not claim external validation, production deployment, or completed empirical findings unless the project-specific record states those facts.

## Flagship Proof Builds

### Claim-to-scope hardening proof

- Claim: Quality Review should show how a project claim is narrowed until it matches evidence, status, audience, and readiness.
- Input: one public project claim, supporting artifacts, scope limit, audience, and status record
- Operation: compare the claim against evidence, implementation state, source support, and release risk
- Expected output: claim hardening decision: keep, narrow, source, defer, or remove
- Verification check: select one overbroad claim and show the exact reason it is kept, narrowed, deferred, or removed

### Completion-language restraint proof

- Claim: A project should not be called built, complete, validated, published, or deployed unless the proof state supports that word.
- Input: status sentence, component list, evidence file, runtime proof, and external-review status
- Operation: map public wording to proof tier and block stronger language when evidence is missing
- Expected output: approved public wording plus blocked stronger alternatives
- Verification check: test one status paragraph and show which words are permitted, which are blocked, and why
