# Quantitative Capstone — quantitative arm of the AI-corruption capstone study

## Public Pitch

**What it is.** A quantitative future-work branch of the submitted qualitative capstone, not the submitted capstone itself.

**Distinctive mechanism.** It turns the Cognitive Firewalls question into controlled condition logic: a fifty-scenario, two-dose, three-protocol matrix with clean versus contaminated evidence, D0/D1 document pairs, DP1/DP2 packets, Scenario Factory materials, Blue/Red role architecture, DV-B contamination-handling logic, Baseline/ACH/CHAMPS protocol comparisons, scorer-skeleton logic, reliability checks, prompt and provenance records, and bounded interpretation rules.

**Why it matters.** It shows experimental-design capacity without pretending that unrun or reliability-limited quantitative machinery has already produced validated findings; floated runset options remain future-work options until adopted and executed.

## Blueprint

### Vision

A quantitative future-work branch of the submitted qualitative capstone, not the submitted capstone itself.

### Mechanism

It turns the Cognitive Firewalls question into controlled condition logic: a fifty-scenario, two-dose, three-protocol matrix with clean versus contaminated evidence, D0/D1 document pairs, DP1/DP2 packets, Scenario Factory materials, Blue/Red role architecture, DV-B contamination-handling logic, Baseline/ACH/CHAMPS protocol comparisons, scorer-skeleton logic, reliability checks, prompt and provenance records, and bounded interpretation rules.

### Built components

Built quantitative foundation includes the Scenario Factory, the fifty-row and thirty-five-column master scenario registry, a later fifty-workbook D0/D1 packet corpus, one hundred evidence-packet references, ten DP1 analyst-facing D0/D1 DOCX packet pairs, ground-truth keys, contamination keys, poison-spec allowed-difference controls, safety scans, integrity manifests, the Scorer Skeleton, schema validation, deterministic scoring, paired D0/D1 differential analysis, diff-gate enforcement, results packaging, DP1/DP2 execution doctrine, gate criteria, role cards, prompt packs, provenance templates, and the March calibration/gate record.

### Source use

The run kit remains the quantitative authority, but the Capstone Quantitative Monograph adds a stricter reconciliation layer. Scenario Factory defines the code-facing scenario universe; Scorer Skeleton defines scoring and differentials; the master registry, crosswalk, and vector map define scenario authority and coverage; packet workbooks preserve the D0/D1 stimulus layer; source-control workbooks govern bibliography, FIMI, non-English, and scenario-claim binding; and the March methodology and gate files define the actual inferential ceiling. Committee and defense files translate the system into prose, but do not override the run/gate evidence.

### Plan

Next plan: make the page read as a governed quantitative workbench with an explicit stop-line. Show the full 300-run design grammar, the DP1 built subset, the human calibration slice, the DV-A versus DV-B reliability split, the NO_GO decision where confirmed, and the registry drift between canonical scenario authority and Factory v5. Treat floated pilot runsets as archival branch options, not as the adopted plan or an unfinished obligation.

### Status

Current status: the quantitative branch is stronger as a built system than a generic capstone summary suggests. It preserves scenario authority, packet generation, packet workbooks, DP1 exports, Blue/Red role architecture, prompt and provenance records, source-control governance, deterministic factory and scorer code, calibration rows, and a formal March 6 reliability gate. It is weaker as a findings claim: DV-B contamination-handling reliability failed the gate, so the system should be presented as built, governed, and stopped at NO_GO rather than as validated efficacy.

### Remaining gaps

Remaining gaps: check every March-gate claim against the underlying gate memo and calibration workbook before making it prominent; reconcile the original Baseline/ACH/CHAMPS naming with any later execution vocabulary; expose registry drift without making it look like error concealment; do not convert floated runset options into adopted status; complete valid dual-human DV-B coding and inter-rater reliability review; reopen inference only after reliability clears the governing floor; and avoid effect-size, model-family, external-validation, or operational-efficacy claims until the gate is cleared.

## Maintenance Rule
