# AI Extraction Defense — five-layer architecture protecting proprietary AI models from being extracted, copied, or distilled by adversaries

## Public Pitch

**What it is.** An extraction-defense architecture for protecting valuable analytical models from copying, distillation, or downstream imitation.

**Distinctive mechanism.** It treats model extraction as a counterintelligence problem: L1 behavioral shaping, L2 dynamic controls, L3 canary narratives, L4 tarpit/degraded-environment design, access segmentation, query-behavior signals, output fingerprinting, latency injection, logical loops, and post-hoc extraction detection work together.

**Why it matters.** The distinctive move is that defense continues after the interaction; a canary can survive paraphrase, while tarpit and ATLAS-style threat-model thinking make extraction costly, observable, and strategically misleading.

## Blueprint

### Vision

An extraction-defense architecture for protecting valuable analytical models from copying, distillation, or downstream imitation.

### Mechanism

It treats model extraction as a counterintelligence problem: L1 behavioral shaping, L2 dynamic controls, L3 canary narratives, L4 tarpit/degraded-environment design, access segmentation, query-behavior signals, output fingerprinting, latency injection, logical loops, and post-hoc extraction detection work together.

### Built components

The architecture is fully specified — a substantial design document, a self-imposed red-team audit nearly as long, an ATLAS threat-model mapping, a bibliography, and a doctrine translation that fully maps CI tradecraft to ML security. The prototype directories are intentionally empty. The project's own claim discipline is what enforces this: zero implemented controls is preferable to stub controls, because "stub-but-claimed-implemented" is exactly the failure mode the discipline forbids. The honest open-research problem the project pages in its own README: "Designing semantic watermarks that are both detectable and imperceptible to the adversary is an open research problem." Until that problem has a defensible answer, L3 is design-only by intellectual necessity, not by neglect.

### Source use

Source use is organized by role: method authorities, domain evidence, technical background, implementation evidence, and claim limits. The page should show why each source matters; direct quotations and stronger factual claims still require exact passage anchors before publication.

### Plan

Layer-by-layer, each layer surviving independent red-team review before the next one starts. The slow cadence is by design — the discipline that refuses implemented-without-prototype refuses equally to promote a layer that has not been adversarially reviewed. Step 01 Threat-model lock Attacker-capability assumptions finalized before any code is written. Step 02 L1 session classifier Behavioral classifier with real quantitative thresholds and ROC analysis; survives independent adversarial review before L2 begins. Step 03 L2 honeytoken engine Liability framework resolved for false-positive injection — a legitimate user might act on a fabricated capability, and the engine has to bound that. Step 04 L3 canary narrative Narrative engine wired against the locked threat model. Step 05 L4 tarpit Tarpit deployment with resource-exhaustion accounting. Step 06 L5 forensic attribution Variant-generation key infrastructure that makes attribution actually traceable end-to-end.
