# Decision Advantage System — operating architecture for intelligence and security organizations

## Public Pitch

**What it is.** A decision-advantage operating architecture for intelligence and security organizations.

**Distinctive mechanism.** It binds Living Briefs, Provenance Fabric, Stop-the-Line Charter, and an operating environment across Direction S1 Strategic Foresight, Sensemaking S2, Analytic Craft, S3 Sensor-to-Sensemaking, Assurance S4, Post-Quantum and Crypto Agility, S6 Software Supply-Chain Assurance, S7 Cognitive Security, influence operations, interagency work, coalition interoperability, feedback, human roles, machine roles, and assurance gates.

**Why it matters.** It shows organizational design skill: the project is not a tool alone, but a proposed operating architecture for decision advantage across direction, sensing, analysis, product integrity, and assurance.

## Blueprint

### Vision

A decision-advantage operating architecture for intelligence and security organizations.

### Mechanism

It binds Living Briefs, Provenance Fabric, Stop-the-Line Charter, and an operating environment across Direction S1 Strategic Foresight, Sensemaking S2, Analytic Craft, S3 Sensor-to-Sensemaking, Assurance S4, Post-Quantum and Crypto Agility, S6 Software Supply-Chain Assurance, S7 Cognitive Security, influence operations, interagency work, coalition interoperability, feedback, human roles, machine roles, and assurance gates.

### Built components

The v0.1 architecture is complete enough to stand up a team without returning to the source conversation. Every service has a contract, every ritual has a runbook, every template is filled out with a worked example, and the 90-day standup roadmap has named exit criteria — by day 90 the organization must produce one Living Brief with change notes, one Counter-Brief, one Provenance Envelope, one assurance note, one Decision Log entry, and one ritual-derived architecture change. The package is explicitly generic . Org-tailored authorities, budget, procurement, jurisdictional classification rules, and legal review are flagged as implementation customizations rather than handoff prerequisites. The architectural warning is part of the package: do not stand up twelve services as separate bureaucratic islands; in small organizations one team may wear multiple service hats.

### Source use

Source use is organized by role: method authorities, domain evidence, technical background, implementation evidence, and claim limits. The page should show why each source matters; direct quotations and stronger factual claims still require exact passage anchors before publication.

### Plan

The forward path is org-tailoring. The architecture is built to be refactored in use, not adopted as a fixed reference, so the next moves are about installing it inside a real organization and letting that organization's actual constraints shape the implementation. Step 01 Org tailoring Authorities, budget, procurement, jurisdictional classification, and legal review filled in for the receiving organization. Step 02 Minimum nucleus standup S2 + S3 + S4 + Living Briefs + Provenance Fabric + Stop-the-Line Charter spun up; one mirror-brief cycle and one assumptions funeral run within 14 days. Step 03 First Attack Week First quarterly adversarial stress of the flagship AI workflow — must change something concrete or the ritual fails its own test. Step 04 v0.2 architecture revision First architecture revision based on what the rituals surfaced — the move from generic v0.1 to organization-specific v0.2.
