# Development Process — how projects move from idea to built, reviewed work

## Public Pitch

**What it is.** A public explanation of how projects are organized and developed across the portfolio.

**Distinctive mechanism.** It maps the route from idea generation to source gathering, project room creation, instrument selection, review, release, and status tracking.

**Why it matters.** It helps a first-time reader understand that the site is not a pile of pages but a project-development system with repeatable stages.

## Blueprint

### Vision

A public explanation of how projects are organized and developed across the portfolio.

### Mechanism

It maps the route from idea generation to source gathering, project room creation, instrument selection, review, release, and status tracking.

### Built components

Built components include project grouping, navigation, room/subroom inventories, status language, release gates, source-use layers, and reviewer-facing claim limits in the generated professional package.

### Source use

Sources are used operationally: governance standards define required project files, room-fidelity records define presentation constraints, and source lineage ledgers define whether a page can claim passage-level support.

### Plan

Plan: keep the planning page above major projects, use plain professional labels, separate project categories by function, and require every project page to state vision, mechanism, What is built, source use, status, and remaining gaps.

### Status

Status: the current package has a native project index and universal reviewer/source sections. The remaining editorial task is to turn every extracted port into authored public prose instead of relying on selected project Evidence Sections.

### Remaining gaps

Remaining gaps: deeper page-by-page source reconciliation, more concise public summaries for dense machinery, and stronger differentiation between live systems, buildable handoffs, internal methods, and archival reference packets.

## Maintenance Rule

Update this blueprint before strengthening public claims on the website. Do not use word count, module count, or status percentage as a substitute for concrete project substance.
