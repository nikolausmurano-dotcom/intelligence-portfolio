# AI Rigor Detection - framework that detects performative rigor in AI-generated analysis; publication status is separated from validation status - Proof Fixture Kit

## Purpose

This kit defines the first concrete proof run needed to move the project from operational contract toward captured working evidence.

## Representative Input

One claim cluster, source cluster, or method section from the research project.

## Operation Under Test

Map the claim to evidence, identify method or scope limits, add counterargument handling, and produce a reviewer-facing section packet.

## Expected Output

A claim-evidence-method packet with source state, uncertainty boundary, and revision decision.

## Acceptance Tests

- Input fixture is explicit and repeatable.
- Operation under test is specific to this project, not generic portfolio prose.
- Expected output is inspectable as a file, page state, table, packet, or export.
- Acceptance criteria are observable and falsifiable.
- Claim boundary prevents stronger claims than the evidence supports.
- At least one load-bearing source role is verified or explicitly marked as awaiting human close reading.

## Falsifier

The fixture fails if the project produces prose without claim-to-source support or if scope conditions are missing.

## Evidence To Capture

- input fixture
- operation trace
- output artifact
- acceptance-test result
- source-verification note where applicable
- claim-boundary note
- reviewer or outside-use evidence before stronger public claims

## Next Build Step

Add one implementation scenario: policy problem, decision rule, control point, abuse case, measurable criterion, and revision trigger.

## Claim Boundary

This proof fixture is a build instrument. It does not by itself prove final operational standing until executed and captured.
