AI Extraction Defense - five-layer architecture protecting proprietary AI models from being extracted, copied, or distilled by adversaries

A concise project summary for professional review, drawn from the maintained project record.

What this does

An extraction-defense architecture for protecting valuable analytical models from copying, distillation, or downstream imitation.

Record note: project purpose

How it works

It treats model extraction as a counterintelligence problem: L1 behavioral shaping, L2 dynamic controls, L3 canary narratives, L4 tarpit/degraded-environment design, access segmentation, query-behavior signals, output fingerprinting, latency injection, logical loops, and post-hoc extraction detection work together.

Record note: working mechanism

Why it matters

The distinctive move is that defense continues after the interaction; a canary can survive paraphrase, while tarpit and ATLAS-style threat-model thinking make extraction costly, observable, and strategically misleading.

Record note: professional value

What is already there

The architecture is fully specified — a substantial design document, a self-imposed red-team private support file nearly as long, an ATLAS threat-model mapping, a bibliography, and a doctrine translation that fully maps CI tradecraft to ML security.

Record note: current status

The prototype directories are intentionally empty.

Record note: current status

The project's own claim discipline is what enforces this: zero implemented controls is preferable to stub controls, because "stub-but-claimed-implemented" is exactly the failure mode the discipline forbids.

Record note: current status

The honest open-research problem the project pages in its own README: "Designing semantic watermarks that are both detectable and imperceptible to the adversary is an open research problem." Until that problem has a defensible answer, L3 is design-only by intellectual necessity, not by neglect.
